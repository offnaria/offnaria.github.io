前回の続き。

今回から、CFU-Playgroundのセットアップに取り組もうと思う。

CFU-Playgroundについて改めてまとめると，「VexRiscVプロセッサのパイプラインに，機械学習系の計算で使いがちな処理を高速に実行する回路を拡張し，拡張命令とその対応付け，さらに各種FPGA向けビットストリーム生成まで自動で行うアプリケーション」といったところである．

環境

CFU-Playgroundのように、日本語のドキュメントが少ないアプリケーションのセットアップには失敗がつきものだと思う。 したがって、環境破壊を繰り返しても許されるようにDockerを用いて準備を行いたい。

また、Xilinx FPGA向けの環境として、CondaのパッケージであるSymbiFlowを用いることが提案されている(Option 4a)。 今回はSymbiFlowを使ってみようと思うので、Vivado環境がすでに存在している場合は参考にならない可能性があることを断っておく。

コピペ用

以下のコマンドたちを入力すれば，とりあえず動くようになると思われる．

apt update
apt install -y sudo patch g++ zlib1g-dev
cd ~
git clone https://github.com/google/CFU-Playground.git
cd CFU-Playground
scripts/setup -ci
make install-sf
cd ~
wget https://static.dev.sifive.com/dev-tools/freedom-tools/v2020.08/riscv64-unknown-elf-gcc-10.1.0-2020.08.2-x86_64-linux-ubuntu14.tar.gz
tar xvfz riscv64-unknown-elf-gcc-10.1.0-2020.08.2-x86_64-linux-ubuntu14.tar.gz
echo "export PATH=$PATH:$HOME/riscv64-unknown-elf-gcc-10.1.0-2020.08.2-x86_64-linux-ubuntu14/bin" >> ~/.bashrc
source ~/.bashrc

以下は，これに至るまでの道のりである．不要な場合は，テスト実行まで読み飛ばしても問題ないと思われる．

セットアップ

基本的に、セットアップはSetup Guideの通りに行うが、引っかかった個所がいくつかあったため、それと対策を共有したい。

Anaconda環境の構築

Docker Hubのページにある通り、pullしてrunするだけ。

docker pull continuumio/anaconda3
docker run -i -t continuumio/anaconda3 /bin/bash

CFU-Playgroundリポジトリのclone

cloneするだけ。一応ホームディレクトリに移動していることに注意されたい。

cd ~
git clone https://github.com/google/CFU-Playground.git

本当はroot以外のユーザを作っておくべきなのだろうけれど、面倒なのでそのままやる。

セットアップスクリプトの実行

ここで結構ハマった。まずは普通にスクリプトを実行してみる。

cd CFU-Playground
./scripts/setup

すると、sudoがないと怒られるので、インストールする。このとき、リポジトリ情報を更新しておかないと、インストールが上手くいかない可能性がある。そのため、あらかじめ更新も行う。

apt update
apt install sudo

再度スクリプトを実行すると、インストールされていないパッケージたちが表示されるので、yを2回入力してインストールする。 なお、インストールされていないパッケージたちは次のようになった。

gcc
Missing packages: gcc make cmake openocd expect ccache verilator libevent-dev libjson-c-dev ninja-build libusb-1.0-0-dev libftdi1-dev

今度はpatchがないと怒られるので、やはりこれもインストールする。

apt install patch

三度目の正直に懸けて、再びスクリプトを実行する。ここで、スクリプトの第一引数として-ciを渡しておくと、パッケージのインストール時にyの入力を求められない。詳しくは、この記事の最後に書く。

RISC-VツールチェーンとVivadoが見つからないという警告の後，セットアップが完了する．

SymbiFlow関連のインストール

CFU-PlaygroundのルートにMakefileが用意されているので，それをそのまま利用する．

make install-sf

インストールが完了したら，SymbiFlowの環境に入れるかどうか確認する．make enter-sfでサブシェルが起動し，exitで抜けられる．

テスト実行

テンプレートプロジェクトをビルドすることにより，インストールした者たちが正常に動作しているかどうかを検証する．Step 6ではArty A7-35T向けにビットリーム生成・書き込みするところまで行っているようだが，手元にボードがあるわけではないため，Verilatorを使ったシミュレーションを行う．

cd CFU-Playground/proj/proj_template
make clean
make PLATFORM=sim load # Verilatorを用いたシミュレーションを行うためのパラメータ

これを実行したところ，g++が見当たらないとのこと．先ほどPATHを通した~/riscv64-（略）/binの中にriscv64-unknown-elf-g++が存在してはいるのだが，なぜこのエラーが出てしまうのかはよく理解できていない．

おそらく，ここで要求されているg++はRISC-VのクロスコンパイラではなくVerilatorで使用するコンパイラなのではないかと考えられる． おとなしくインストールしよう．

apt install g++

セットアップスクリプトの-ciオプションについて

シェルスクリプト初心者なので、スクリプトに書かれていることの意味を調べながら理解したことを書く。 中・上級者は既に知っていることばかりであると思われる。

セットアップスクリプト（commit b6c9151）を眺めてみると、18～47行目ではインストールされていないパッケージたちを把握していき、49,50行目でそれらを出力していることが分かる。

続いて、52～74行目では、apt installにyを付けるかどうかの処理を行っている。 以下に、コメントの形で解説を行う。

in_ci=0
# ifの条件式には略式testコマンドが使われている
if [ $# -gt 0 ] ; then      # 引数の総数($#)は0より大きい(greater than)か？
  if [ $1 == "-ci" ] ; then # 第一引数($1)は"-ci"か？
    in_ci=1
  fi
fi

if [ ${#missing[@]} -gt 0 ]; then                # missingの個数(${#missing[@]})は0より大きいか？
    echo "Missing packages: ${missing[@]}" >&2   # ${missing[@]}でmissingの中身を列挙できる
    if which apt >/dev/null; then                # aptが/dev/nullの下にあるか（文字列比較？なんでtestじゃないんだろう？）
        if [ $in_ci -gt 0 ]; then                # in_ci>0 ⇐ 第一引数が"-ci"
            sudo apt install -y "${missing[@]}"  # "y"付きでapt install
        else
            echo -n "Install [y]/n? " >&2        # yの入力を要求
            answer=$(read)
            if [ "$answer" == "y" -o -z "$answer" ]; then
                sudo apt install "${missing[@]}" # apt installもyの入力を要求するので2回要求される羽目になる
            fi
        fi
    else
        exit 1
    fi
fi

（参考：if 文と test コマンド | UNIX & Linux コマンド・シェルスクリプト リファレンス）
